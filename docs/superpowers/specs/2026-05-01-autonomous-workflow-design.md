# Autonomous Workflow Engine — Design Spec

Captured: 2026-05-01

---

## Overview

A modular, spec-driven workflow orchestration skill for ClaudeSkills that:
- Lives at `skills/ai-agents/autonomous-workflow/`
- Bootstraps a runnable runtime scaffold on first invocation
- Integrates one-way with AgenticOS as an event broadcaster
- Extends the ralph loop pattern with phase isolation and hybrid multi-role voting

## Section 2 — SKILL.md Entry Logic

### Trigger Conditions

The skill triggers on any of:
- "autonomous workflow", "workflow engine", "run this end to end"
- "execute phases", "ralph loop", "orchestrate this"
- Presence of `spec.md`, `phases.json`, or `state/workflow_state.json` in working directory
- Explicit invocation with `/autonomous-workflow`

### Adaptive Resume Logic (D+B)

Priority order on invocation:

1. `--from=<phase>` flag present → jump to specified phase regardless of any state (explicit user intent wins)
2. `state/workflow_state.json` exists → resume from last incomplete phase
3. `phases.json` exists (no state) → skip brainstorm+planning, enter execution
4. `spec.md` exists (no phases.json) → skip brainstorm, enter planning
5. Nothing found, no flag → start at brainstorm

### Argument Interface

```
/autonomous-workflow [--from=brainstorm|planning|execution|verification]
                     [--no-vote]
                     [--agenticos-url=<url>]
                     <task description or path to spec>
```

| Flag | Default | Purpose |
|------|---------|---------|
| `--from` | auto-detected | Override resume point |
| `--no-vote` | off | Disable voting gates (single-agent simulation only) |
| `--agenticos-url` | `http://localhost:8000` | AgenticOS state bus URL |

## Section 3 — Ralph Loop Extension

### Base Contract (from ralph/ultrawork)

- PRD-driven persistence: work continues until all user stories have `passes: true`
- Story-by-story iteration with reviewer sign-off before completion
- Progress tracked in `state/progress.txt` across iterations

### Extensions

| Feature | Base Ralph | This Extension |
|---------|-----------|----------------|
| PRD generation | Manual | Auto-generated by planning phase via GSD decomposition |
| Story granularity | Task-level | Phase-level (each story = one workflow phase) |
| Voting gate | None | Hybrid: simulated for routine, parallel subagents for high-stakes |
| Context isolation | Single session | Each phase = isolated headless agent context |
| State broadcast | None | `agenticos_push.py` fires `workflow.phase_complete` events |

### Phase Definitions

| Phase | Purpose | Method | Output |
|-------|---------|--------|--------|
| `brainstorm` | Clarify intent and requirements | GStack-style 4-role simulation (CEO, Engineer, Designer, Security) in single context | `spec.md` |
| `planning` | Decompose spec into executable phases | GSD-style phase decomposition | `phases.json` |
| `execution` | Implement each phase with TDD | Superpower-style TDD — write tests first, verify red, implement, verify green | `code + tests` |
| `verification` | Validate correctness, performance, security | Automated review agents + test runner | `verification_report.md` |

### Hybrid Voting Protocol

**Simulated (single context):** routine implementation decisions, minor architectural choices, style/naming decisions

**Parallel subagents (4 concurrent):** architecture decisions that affect 2+ phases, security boundary definitions, integration contracts with external systems (AgenticOS, MCP servers), any decision where roles would have conflicting incentives

Voting procedure:
1. Spawn Engineer, Architect, PM, Security agents concurrently with role-scoped prompts
2. Each returns: `{ "vote": "pass|fail", "justification": "...", "concerns": [...] }`
3. Majority (3/4) required to pass; tie goes to Architect
4. Result + justifications logged to `state/votes.json`
5. Blocking concern from Security always escalates regardless of majority

### Phase Isolation

Each phase runs in its own headless agent context:
- No shared memory between phases except through explicit state files
- Context window starts clean per phase
- Inputs passed as file paths, not inline content
- Prevents context degradation across long-running workflows

## Section 4 — AgenticOS Integration

### Coupling Model

**One-way push only.** The skill owns `state/workflow_state.json`. AgenticOS is a subscriber, not an authority. The skill functions completely if AgenticOS is unavailable — all push calls are fire-and-forget with silent failure.

### Events Emitted

```json
{ "event": "workflow.started",        "workflow_id": "uuid", "task": "...",          "timestamp": "ISO8601" }
{ "event": "workflow.phase_started",  "workflow_id": "uuid", "phase": "brainstorm",  "timestamp": "ISO8601" }
{ "event": "workflow.phase_complete", "workflow_id": "uuid", "phase": "planning",    "timestamp": "ISO8601" }
{ "event": "workflow.vote_cast",      "workflow_id": "uuid", "decision": "...",      "result": "pass|fail", "voters": ["Engineer","Architect","PM","Security"] }
{ "event": "workflow.complete",       "workflow_id": "uuid", "phases_run": 4,        "timestamp": "ISO8601" }
{ "event": "workflow.failed",         "workflow_id": "uuid", "phase": "execution",   "reason": "...",       "timestamp": "ISO8601" }
```

### AgenticOS Endpoint

POSTs to `/events` on the AgenticOS FastAPI state bus (default: `http://localhost:8000/events`).

If the `/events` route does not exist in the current AgenticOS version, `agenticos_push.py` falls back to `/state` with a `workflow_event` key. Route detection happens once on startup and is cached.

### React HUD Visibility

AgenticOS can display active workflow progress by subscribing to `workflow.*` events. No changes required to the skill — the HUD consumes events already in the bus. A `WorkflowStatusPanel` component can be added to the frontend in a future session.

## Section 5 — Security Constraints

- No plaintext secrets in any skill file or state file
- AgenticOS URL sourced from `--agenticos-url` flag or `AGENTICOS_URL` env var only
- `bootstrap.py` validates all file paths before writing (no path traversal)
- Voting subagents receive role-scoped prompts only — no full context passed
- `state/` directory is gitignored to prevent leaking workflow state

---

## Section 6 — Performance Constraints

- Each phase context starts under 20% window usage (phase isolation enforces this)
- `phases.json` is the only cross-phase dependency — kept under 50 lines
- Parallel voting only for high-stakes gates (estimated: ≤2 per workflow)
- `agenticos_push.py` uses async HTTP with 2s timeout — never blocks execution
